## Context

Issue #151 ("docker quick start not working"): a self-hoster followed the README Docker quick-start, the containers report healthy, the DB is listening, but `http://localhost:3000` does not respond. The app container logs `Local: http://...:80`. A contributor identified the cause in-thread (and it got community confirmation): `docker-compose.yml` forwards the operator's whole `.env` into the `app` container via `env_file: .env`, and the published image's Next.js standalone `server.js` reads `process.env.PORT` for its bind port. A stray `PORT` in `.env` (e.g. `PORT=80`) makes the app bind to that port inside the container while compose still maps host `3000 -> container 3000`, so the publish points at a dead port.

`.env.example` ships without `PORT`, which is why `cp .env.example .env` "just works" - the bug only bites operators who add a generic `PORT` to their `.env`.

## Goals / Non-Goals

**Goals:**
- Following the documented quick-start reliably yields a reachable app on the published host port, even if the operator's `.env` contains an unrelated `PORT`.
- The container's internal listen port matches the container side of the `ports` mapping deterministically.
- Document the behavior so operators understand `PORT` in `.env` does not change the published port.

**Non-Goals:**
- No change to the application's runtime/port-selection code (`server.js` is generated by Next.js; we control the env, not the binary).
- Not making the published port itself configurable via `.env` in this change (out of scope for the bug; would need coordinated host+container mapping and is a separate enhancement).
- No change to the `db` service or to the developer compose flow beyond the app port pin.

## Decisions

- **Pin `PORT=3000` via `environment:` on the `app` service rather than telling users to remove `PORT` from `.env`.** In Docker Compose, a value set in a service's `environment:` overrides the same key sourced from `env_file:`. Pinning makes the quick-start robust by construction instead of relying on every operator keeping their `.env` clean. The contributor's "remove PORT from .env" advice fixes one operator's machine; pinning fixes it for everyone and is self-documenting in the compose file.
  - Alternative considered: drop `env_file:` and enumerate only the vars the app needs. Rejected - higher blast radius (would have to track every consumed var) and changes how operators configure the app.
  - Alternative considered: change the `ports` mapping to read `${PORT}`. Rejected - it makes the failure mode worse/more confusing and still lets a stray container `PORT` diverge from the host mapping.
- **Keep the container port fixed at 3000** (matching `EXPOSE 3000`, the `3000:3000` mapping, and `NEXTAUTH_URL=http://localhost:3000`). Operators who want a different *host* port change the left side of the `ports` mapping; the container side stays 3000.

## Risks / Trade-offs

- [An operator who deliberately set `PORT` in `.env` expecting it to remap the container] -> They were already broken (publish mismatch); the README note explains the new, correct way to remap (edit the `ports` host side). Net behavior is strictly more correct.
- [Compose precedence assumption is wrong] -> Mitigated: docker-compose documents that `environment:` overrides `env_file:`; the spec/test asserts the compose file contains the explicit `environment: PORT=3000` on the app service and that the container side of the publish is 3000, so a regression (e.g. someone deletes the pin) is caught.

## Migration Plan

Pure compose + docs change. Operators pull the new image / compose file and `docker compose up -d` as usual; no data migration, no rollback concern (revert the commit if needed). Existing working setups are unaffected because they had no conflicting `PORT`.
